Architecture

Le serveur Flask centralise les données envoyées par les clients IoT selon trois protocoles de transmission différents.

Structure du projet

IoT_project/ ├── db/ │ └── iot_monitoring.db # SQLite · créée automatiquement ├── data/ │ ├── netlog_s1.csv # Logs réseau — stratégie S1 │ ├── netlog_s2.csv # Logs réseau — stratégie S2 │ ├── netlog_s3.csv # Logs réseau — stratégie S3 │ └── dataset.csv # Jeu de données simulé (entrée) └── src/ ├── server.py # Serveur Flask · POST /data ├── client_common.py # load_dataset · build_sample · post_payload ├── client_s1.py # Envoi individuel (1 sample / requête) ├── client_s2.py # Envoi par batch (BATCH_SIZE = 20) └── client_s3.py # Filtrage événementiel (presence == 1)
// 02 — protocoles

Les 3 stratégies de transmission

Chaque stratégie représente un compromis différent entre réactivité, efficacité réseau et charge serveur.

S1

Envoi individuel

Chaque sample est transmis immédiatement, un par un. Latence minimale, volume de requêtes maximal.

n_samplesToujours 1
requêtes1 par sample
sortienetlog_s1.csv
usageTemps réel critique
S2

Envoi par batch

Les samples s'accumulent jusqu'à atteindre BATCH_SIZE = 20, puis sont envoyés en une seule requête.

n_samples≤ 20 par requête
requêtes÷ 20 vs S1
sortienetlog_s2.csv
usageBande passante limitée
S3

Filtrage événementiel

Seuls les samples avec presence == 1 (activité détectée) sont transmis. Les événements "vides" sont ignorés.

n_samplesToujours 1
requêtes% présence uniquement
sortienetlog_s3.csv
usageAlerting / anomalie

Comparaison des patterns de transmission (8 samples, 3 avec presence=1)

// 03 — mise en route

Démarrage rapide

Le serveur doit toujours être lancé avant les clients. Les répertoires db/ et data/ sont créés automatiquement.

1

Installer les dépendances

Python 3.7+ requis. Flask pour le serveur, pandas pour la lecture du dataset, requests pour les appels HTTP.

bash pip install flask pandas requests
2

Préparer le dataset

Placer un fichier CSV dans data/dataset.csv contenant les colonnes timestamp, person_id et presence (0 ou 1).

3

Démarrer le serveur Flask

Ouvrir un terminal dédié et laisser tourner le serveur.

bash cd src
python server.py
# * Running on http://0.0.0.0:5000
4

Exécuter les clients (dans d'autres terminaux)

Lancer chaque client séquentiellement pour générer les trois fichiers de logs.

bash cd src
python client_s1.py # → data/netlog_s1.csv
python client_s2.py # → data/netlog_s2.csv
python client_s3.py # → data/netlog_s3.csv
⚠
Attention : les fichiers CSV sont écrasés à chaque exécution d'un client. Pour conserver l'historique, remplacer le mode "w" par "a" dans l'ouverture du fichier.
// 04 — interfaces

API Serveur

Le serveur expose un unique endpoint POST /data qui accepte des samples individuels ou en liste.

Format de la requête JSON :

json {
  "strategy": "S1", // "S1" | "S2" | "S3" | "UNK"
  "samples": [ // objet unique ou liste
    {
      "timestamp": "2024-01-15T08:32:11",
      "person_id": "CARD_042",
      "presence": 1
    }
  ]
}

Réponse du serveur :

json {
  "status": "ok",
  "t_recv": "2024-01-15T08:32:11.452319" // utilisé pour calculer la latence
}

// 06 — exploitation

Analyse des résultats

Comparer les trois stratégies à partir des fichiers CSV générés.

python import pandas as pd

# Charger les logs
s1 = pd.read_csv("../data/netlog_s1.csv")
s2 = pd.read_csv("../data/netlog_s2.csv")
s3 = pd.read_csv("../data/netlog_s3.csv")

# Calculer la latence (en secondes)
for df in [s1, s2, s3]:
    df["latency"] = pd.to_datetime(df["t_recv"]) - pd.to_datetime(df["t_send"])
    df["latency_ms"] = df["latency"].dt.total_seconds() * 1000

# Comparaison des indicateurs clés
for name, df in {"S1": s1, "S2": s2, "S3": s3}.items():
    print(f"[{name}] requêtes={len(df)} "
          f"latence_moy={df['latency_ms'].mean():.1f}ms "
          f"payload_moy={df['size_bytes'].mean():.0f}B")

À mesurer sur S1

Latence brute, charge réseau maximale, overhead par requête.

À mesurer sur S2

Gain de volume de requêtes, latence accrue, efficacité payload.

À mesurer sur S3

Réduction de trafic (% presence=1), latence vs S1, données manquantes.
